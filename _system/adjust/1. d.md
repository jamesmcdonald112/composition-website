# Process Notes — Things to Improve

## 1. Layout System
Claude needs to apply layout systematically — containers, max-widths, 
gutters, and section spacing — consistently across every component.

**Fix:** Add `view _system/LAYOUT-REFERENCE.md` to every build prompt.
The layout reference document defines the exact rules. Once it is in 
context, Claude applies them without being told each time.

---

## 2. Component Direction — Be More Descriptive
Vague prompts produce generic output. Claude needs a specific layout 
concept for every component before it writes any code.

**Fix:** Use the formula from COMPONENT-DIRECTION-GUIDE.md every time:
- Describe the layout concept in plain English
- Specify font-family, font-size, and color tokens per element
- Describe hover behaviour explicitly
- State GSAP requirements if needed
- List hard rules at the bottom

The testimonial component is the benchmark. It worked because the 
concept was clear. Replicate that level of direction for every component.

---

## 3. Pre-Design in Claude Chat Before Building in Terminal
Before running a component session in the terminal, use Claude chat 
(this interface) to work out what the component should look like first.

**Workflow:**
1. Share the relevant config files here in chat
2. Ask Claude chat to suggest a layout concept based on the real content
3. Refine the concept until it feels right
4. Then take that concept into the terminal build prompt

This separates the creative direction (chat) from the execution (terminal),
which is the right division of labour. Terminal Claude executes.
Chat Claude designs.

---

## 4. Build by Component or by Page — Pick One

**By page** (current approach): Build all components for a page in one 
session so the full flow is visible. Good for visual coherence but 
harder to control quality on each individual component.

**By component**: Build one component at a time with full attention. 
Easier to direct and review but slower to see how things work together.

**Recommendation:** Use a hybrid approach:
- Foundation components (Navbar, Footer, Button) → by component
- Page sections → by page, but with layout concepts pre-written 
  for each component before the session starts
- If a component comes out wrong twice → isolate it and build it alone

---

## 5. Use Composition and Utility Classes First

Before writing any component-level CSS, Claude should check `src/styles/compositions.css` and `src/styles/utilities.css` for existing classes that already handle the job — e.g. `.section` for padding-block, `.wrapper` for gutters and max-width, `.stack` for vertical spacing.

Writing component-level CSS that duplicates these creates drift and means constant manual corrections. The rule: reach for a composition or utility class first, only write component CSS for things that genuinely can't be handled by the existing system.

---

## 6. Wrapper Class — Next Project

In the next project, replace the two-class pattern `wrapper wrapper--wide` with a single `wrapper` class set to one fixed size. Avoid the modifier system for the container width — just define one max-width on `.wrapper` and use it everywhere.

## 7. Scroll Entrance Animations — Shared Utility vs Per-Component

Every component currently duplicates the same scroll-entrance pattern (fade up + stagger on ScrollTrigger). Consider consolidating this into one of two approaches:

**Option A — Shared GSAP utility function**
A single `src/scripts/animations.ts` that exports a `fadeInOnScroll(selector, options?)` helper. Each component calls it with its own `data-` attribute selector rather than re-implementing the ScrollTrigger logic from scratch. Reduces boilerplate and keeps easing/duration values consistent across the site.

**Option B — CSS keyframe entrances**
Define a `@keyframes fade-up` in `utilities.css` and an `.animate-on-scroll` utility class. Use an `IntersectionObserver` (one instance, site-wide) to add an `is-visible` class when the element enters the viewport, triggering the animation. No GSAP dependency for simple fades. GSAP stays reserved for complex/scrubbed animations (count-ups, draw-on effects, accordion height tweens).

**Recommendation:** Option B for simple section entrances (header, list stagger). Option A for anything that needs scrubbing, sequencing, or precise timing. Evaluate at the start of the next project.

## 8. Single Source of Truth for Typography and Colour

Heading sizes, eyebrow styles, body text, and colour roles are currently repeated independently inside each component's `<style>` block. This creates drift — the TrustStrip heading was a different size to every other section h2, and only caught by manual audit.

**Fix for the next project:** Define a small set of named typographic roles as utility classes or CSS custom properties at the global level, not inside individual components:

```css
/* e.g. in utilities.css or compositions.css */
.heading-section   { font-size: clamp(2xl, 3.5vw, 4xl); font-weight: light; ... }
.heading-display   { font-size: clamp(4xl, 5.5vw, 5xl); ... }
.eyebrow           { font-size: xs; letter-spacing: widest; text-transform: uppercase; ... }
.body-lead         { font-size: lg; line-height: relaxed; ... }
```

Each component then applies the class rather than restating every property. Colour roles (heading colour, muted text, accent text) should follow the same pattern — one place to change, everything updates.

**For this project:** when a new component is built, check its heading `font-size` against the established scale (`clamp(2xl, 3.5vw, 4xl)` for section h2s) before signing off.
