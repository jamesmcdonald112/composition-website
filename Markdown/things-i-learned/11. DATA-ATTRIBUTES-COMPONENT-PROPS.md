# Passing Data Attributes Through Astro Component Props

How to pass `data-*` attributes into a reusable Astro component cleanly.

---

## The Problem

You want to put a `data-navbar-logo` attribute on the `<a>` tag inside `BrandLogo.astro` so that JavaScript can query it with `querySelector("[data-navbar-logo]")`.

But `BrandLogo` renders the `<a>` — you can't set the attribute from outside unless the component forwards it through.

---

## Why Not Declare It in the Interface

You might think to do this:

```ts
interface Props {
  class?: string;
  "data-navbar-logo"?: string; // ❌ wrong approach
}
```

The quotes are required because hyphens aren't valid in JS identifier names. But this hardcodes a specific data attribute into a reusable component. Every place `BrandLogo` is used now has this prop in its interface — even places that don't need it. If you need a second data attribute later, you'd have to add another entry. The component is no longer generic.

---

## The Right Pattern — Spread the Rest

Pull out what you need (`class`) and spread everything else directly onto the element:

```astro
---
interface Props {
  class?: string;
}

const { class: className, ...rest } = Astro.props;
---

<a
  href="/"
  class:list={["brand-logo", className]}
  aria-label="..."
  {...rest}
>
```

### How `...rest` actually works

`Astro.props` is an object containing everything passed into the component. When you destructure it, you explicitly pull out what you need by name — here, `class` is pulled out and renamed to `className`. The `...rest` syntax means "collect everything that was NOT explicitly named into a new object called `rest`".

So if you pass `<BrandLogo data-navbar-logo="" />`, then:
- `className` = `undefined` (no class was passed)
- `rest` = `{ "data-navbar-logo": "" }`

Then `{...rest}` on the `<a>` tag spreads that object back out as individual HTML attributes:

```astro
<a {...rest}>
// renders as
<a data-navbar-logo="">
```

The component never needs to declare `data-navbar-logo` in the interface. It doesn't need to know about it at all — the attribute just flows through automatically via `rest`.

---

## Why `class` Is the Exception

`class:list` is Astro's way of merging multiple class sources. If you spread `class` along with `...rest`, it would conflict with `class:list`. So you pull it out first, handle it via `class:list`, and spread the rest.

---

## Passing It In

```astro
<BrandLogo data-navbar-logo="" />
```

No special prop declaration needed. The attribute goes straight through to the `<a>` tag.

The value `""` is used because JavaScript only checks whether the attribute **exists** — `hasAttribute("data-navbar-logo")` or `querySelector("[data-navbar-logo]")` don't care about the value. Passing `undefined` would cause Astro to omit the attribute entirely.

---

## Used In

- `src/components/ui/BrandLogo.astro` — uses the spread pattern
- `src/components/navbar/Navbar.astro` — passes `data-navbar-logo=""`
- `src/components/navbar/navbar.gsap.ts` — queries `[data-navbar-logo]` for entrance animation
